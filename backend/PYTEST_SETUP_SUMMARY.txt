WHAT IS PYTEST?
    • Testing framework for Python
    • Makes writing tests easy and fun
    • Automatically discovers and runs tests
    • Great error messages

WHY TESTING?
    • Catch bugs early (before production)
    • Refactor code with confidence
    • Document code behavior
    • Required for production-grade code
    • Interview portfolio piece

THE TESTING PYRAMID (what to test):
    
    ▲                           Edge cases, stress tests (1%)
    │                          ╱╲
    │                         ╱  ╲  Integration tests (10-20%)
    │                        ╱    ╱╲     ╱ HTTP + Database
    │                       ╱    ╱  ╲   ╱  ╱ Real requests
    │                      ╱____╱    ╱╲_╱__╱ Real dependencies
    │                              ╱   Unit tests (70-80%)
    └─────────────────────────────╱─────────────────────────
                                Function level
                              No external calls
                           Database mocks (SQLite)
                         Fast (ms, not seconds)

PHASE 1 FOCUS: Building strong test foundation

    ✓ Achieve 80%+ code coverage
"""

# ============================================================================
# TESTING VOCABULARY
# ============================================================================

"""
TEST               = Function that checks if code works
FIXTURE            = Reusable test setup (database, users, tokens)
ASSERTION          = Statement checking code is correct (assert x == y)
MOCK/PATCH         = Fake version of real function for testing
MARK               = Label for categorizing tests (@pytest.mark.unit)
UNIT TEST          = Test one function in isolation
INTEGRATION TEST   = Test whole system (endpoints + database)
COVERAGE           = Percentage of code exercised by tests
PASS/FAIL          = Test succeeded / Test failed
PARAMETRIZE        = Run same test with multiple inputs
"""

    response = client.post("/auth/login", json=login_data)

    assert response.status_code == 200
    assert response.json()["access_token"]
    assert response.json()["token_type"] == "bearer"

TOTAL: Clear, readable, easy to understand!

# ============================================================================
# COMMON ERROR MESSAGES (What They Mean)
# ============================================================================

"""
AssertionError: 422 != 201
    → Validation failed (400/422)
    → Check input data (email format, password, etc.)

AssertionError: assert 'password' in response.json()
    → Expected field missing from response
    → Check response actually has that field

FAILED: fixture "client" not found
    → conftest.py not found or named wrong
    → Check file exists: tests/conftest.py

ModuleNotFoundError: No module named 'app'
    → Not in correct directory
    → Run from: cd backend && pytest

Test timeout / stuck
    → Database not closing properly
    → Check fixture cleanup code
"""

# ============================================================================
# QUESTIONS TO ASK WHILE LEARNING

"""
ABOUT FIXTURES:
    Q: What does @pytest.fixture do?
    A: Creates reusable test setup code
    
    Q: Why use test_db fixture instead of real database?
    A: Faster (in-memory), isolated (fresh DB per test), safe (no cleanup)

ABOUT ASSERTIONS:
    Q: Why use 'assert x == y' instead of 'assertEqual(x, y)'?
    A: Simpler syntax, cleaner error messages, pytest style
    
    Q: What if assertion fails?
    A: Test stops and shows error message

ABOUT MARKS:
    Q: Why use @pytest.mark.unit vs @pytest.mark.integration?
    A: Categorize tests, run subsets, different speeds/purposes
    
    Q: Can I use multiple marks?
    A: Yes! @pytest.mark.unit @pytest.mark.auth def test_something()

ABOUT PARAMETRIZE:
    Q: Why parametrize instead of multiple tests?
    A: Less code duplication, easier to add cases, cleaner
    
    Q: How many cases should I parametrize?
    A: 2-5 cases. More than that = separate test method
"""
